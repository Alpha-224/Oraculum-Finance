# Oraculum-Finance: Data Structure Analysis

Based on the repository's documentation and source code, here is a comprehensive breakdown of the existing data structure for the project.

## 1. Core Architecture: The Single Source of Truth
The project does **not** rely on a relational database (like SQLite or PostgreSQL). Instead, the entire data structure is maintained in a single, append-only JSON file: `data_layer/master_financial_data.json`. 

This file acts as the ultimate ledger and is divided into three primary tables (arrays of JSON objects) along with a metadata block.

```json
{
  "metadata": {
    "created_at": "...",
    "last_updated": "...",
    "source_files": ["..."],
    "record_counts": {
      "transactions": 0,
      "obligations": 0,
      "receivables": 0
    }
  },
  "transactions": [],
  "obligations": [],
  "receivables": []
}
```

## 2. Repository Structure & Team Territories
The project is strictly modularized by team discipline to avoid merge conflicts and enforce separation of concerns.

- `/data_layer/`: Owned by the Data Team. Contains the entire ingestion pipeline (CSV, JSON, OCR parsing), `id_generator.py`, cleaning and validation logic, and the `master_financial_data.json` storage mechanism.
- `/frontend/` (and `/Oraculum-Finance/frontend/`): Owned exclusively by the Frontend Team. Contains the cross-platform Flutter application code.
- `/backend/`: Designed to be owned by the Logic Lead, intended to consume the data generated by the `data_layer` and provide APIs.
- Root files: Contains PRD documents (`Oraculum_Data_Ingestion_PRD_v2.pdf`, `Oraculum_OCR_Ingestion_PRD_v1.pdf`), `repo-instructions.md` for workflow rules, and `instructions.md`.

## 3. Table Schemas
Each of the three tables serves a specific purpose for financial tracking.

### A. `transactions` (Historical Cash Flow)
Records of money that has already moved (income, expenses, transfers).
- **Core Fields**: `transaction_id` (e.g., `txn_YYYYMMDD_uuid4[:8]`), `date`, `description`, `amount`, `type` (income/expense/transfer/starting).
- **Computed Fields**: `running_balance` (recalculated entirely on every merge, never trusted from input), `is_duplicate` (based on a SHA256 fingerprint of date+description+amount).
- **Reference Fields**: `source` (filename or manual entry), `category` (from a standard taxonomy).

### B. `obligations` (Accounts Payable)
Upcoming payments the business owes to vendors.
- **Core Fields**: `obligation_id` (`obl_...`), `vendor`, `description`, `amount` (always positive), `due_date`, `category`.
- **Relationship/Terms Fields**: `is_critical` (boolean), `late_fee`, `grace_days`, `flexibility_score`, `relationship_tier`.
- **Recurrence**: `recurring_payment` (boolean), `periodicity` (e.g., daily/weekly/monthly).

### C. `receivables` (Accounts Receivable)
Expected incoming payments from clients.
- **Core Fields**: `receivable_id` (`rec_...`), `client`, `description`, `amount` (always positive), `expected_date`, `category`.
- **Tracking/Risk Fields**: `probability` (0-1 likelihood of on-time payment), `client_tier`, `payment_terms`, `is_overdue` (dynamically computed comparing `expected_date` to today).
- **Recurrence**: `recurring_payment` (boolean), `periodicity`.

## 4. Transient Data State: OCR Pipeline
The recently added OCR pipeline creates a temporary schema for raw extracted data before it gets formally committed to the ledger.

- **`hand_written_docs` Object**: When an image (receipt/invoice) is parsed by the Gemini Vision API, it is formulated into a `hand_written_docs` object.
- **Fields include**: `doc_id`, `extracted_text`, `merchant`, `total_amount`, `date`, `confidence_score` (0-1), `is_processed`, and `user_verified`.
- **Routing Logic**:
  - **Score > 0.9**: Automatically processed. A formal transaction/obligation/receivable is created, injected into the master JSON, and its new ID is saved back to `hand_written_docs` as `linked_record_id`.
  - **Score 0.7 - 0.9**: Saved as `hand_written_docs` but flagged for `manual_review` by the user in the UI. Not merged into the master JSON.
  - **Score < 0.7**: Rejected entirely.

## 5. Key Data Principles
1. **Append-Only**: Records are only added. In-place updates are not supported by the ingestion pipeline (mistakes must be re-ingested or corrected via a different mechanism).
2. **Deduplication**: Handled robustly via `validators.compute_fingerprint()`. Duplicates are silently skipped.
3. **Running Balances**: Transactions' running balances are strictly computed mathematically (`previous_balance + amount`) upon merge, ensuring absolute consistency.
4. **Independent Territories**: 
   - Backend logic (`/backend`) consumes the JSON.
   - Frontend (`/frontend/lib`) reads the JSON directly and calls manual entry APIs.
   - Data Layer (`/data_layer`) strictly handles the ingestion, cleaning, and writing of the JSON file.
